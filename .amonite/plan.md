# Implementation Plan: 3DS Type

**Spec**: `.amonite/spec.md` · **Principles check**: pass (see bottom)

## Technical context

- **Android app** — Kotlin, Android SDK (min API 21), Gradle build. AccessibilityService captures physical keyboard events; a foreground service dispatches UDP packets. SharedPreferences / DataStore for settings and draft path. No external runtime deps (principle E4).
- **3DS app** — C++, devkitPro / devkitARM toolchain, libctru (HID polling, sockets, SD I/O), citro2d + citro3d (rendering). Targets original 3DS hardware; restricted to 12 standard digital buttons (A/B/X/Y/L/R/Start/Select/D-Up/D-Down/D-Left/D-Right). No ZL/ZR (principle P4).
- **Wire protocol** — Luma3DS InputRedirect, UDP port 4950, 20-byte little-endian HID packet. Bytes 0–3: button bitmask (u32 LE). Bytes 4–19: circle pad, touch, IR, special — all zeroed (neutral) for our use.
- **Shared chord table** — canonical mapping of ~100 chars (printable ASCII 32–126, Backspace, Enter, Delete = 97 entries) to unique 12-bit button bitmasks. Generated by a single script into both a C++ header and a Kotlin object, ensuring round-trip identity. No bitmask 0x000 (idle). Avoid bitmasks that map to START+SELECT (soft-reset in some contexts).
- **Storage** — 3DS: SD card at `/3ds/typewriter/<ISO-timestamp>.txt`. Android: configurable local path (default: app-private files dir), saved per draft session.

**Key design decision — shared chord table**: A code-generation script produces both `chord_table.h` (C++) and `ChordTable.kt` (Kotlin) from a single JSON source of truth. T001 owns this. Both C004 and C002 consume the outputs. The round-trip test in T008 shares the table with T003's encoder test, enforcing byte-exact identity.

**Key design decision — chord timing**: Android sends a key-down packet (bitmask set) followed 16ms later by a key-up packet (bitmask 0x000). The 3DS app reads `hidKeysDown()` (fires once on transition, not on held state) so it sees exactly one character per keystroke.

## gate.live (manual, run after APP builds)

1. Copy `typewriter.3dsx` to 3DS SD card → launch via Homebrew Launcher → confirm boot
2. Enable Luma3DS InputRedirection on 3DS (Rosalina menu)
3. Install APK → grant Accessibility permission → enter 3DS IP → confirm `Connected` in notification
4. Type a short sentence → confirm characters appear on 3DS top screen with no perceptible lag
5. Press Start on 3DS → confirm `.txt` file saved to SD card with correct content
6. Re-launch typewriter.3dsx → confirm draft appears in list → open it → confirm content
7. Confirm Android draft file saved to configured path with correct content

## Risks / complexity

| Risk | Mitigation |
|------|------------|
| Luma3DS HID injection timing: double-char or dropped char if key-up packet timing is off | Ship 16ms key-up delay (one frame at 60fps); make delay configurable in task T011 |
| devkitPro not in nixpkgs stable (25.05) | Pin devkitPro via devkitpro-nix overlay in task envs; document in T007 |
| AccessibilityService may not fire for all keyboard types (OTG vs BT) | Implement InputDevice polling fallback in T003; gate.live test covers both |
| START+SELECT chord collision with system soft-reset | Exclude 0x00C (Start=bit3, Select=bit2) from chord table in T001 generator |
